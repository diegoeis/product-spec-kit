# Quality Checklist - Document Validation

Use this checklist before finalizing any product documentation.

---

## PRD Checklist

### Problem & Goals
- [ ] Problem statement is clear and specific
- [ ] Current pain points are well documented
- [ ] Business opportunity is quantified
- [ ] Target users are clearly defined
- [ ] Success metrics are measurable and realistic

### Requirements
- [ ] All requirements have clear user value
- [ ] Must-have (P0) vs nice-to-have (P2) is clear
- [ ] Non-functional requirements included
- [ ] Technical constraints documented
- [ ] Dependencies identified

### Scope
- [ ] In-scope items are specific
- [ ] Out-of-scope items are explicitly stated
- [ ] Future considerations noted
- [ ] Boundaries are clear

### Stakeholders
- [ ] All key stakeholders identified
- [ ] Review/approval process defined
- [ ] Communication plan exists

### Constitution Principles
- [ ] ✅ User value articulated for every requirement
- [ ] ✅ Success criteria are testable
- [ ] ✅ Document is self-contained
- [ ] ✅ Priorities are clear
- [ ] ✅ Terminology is consistent

---

## Plan Checklist

### Strategy
- [ ] Release philosophy is clear
- [ ] Roadmap shows logical progression
- [ ] Success criteria per release defined

### Structure
- [ ] Releases → Deliveries → Epics hierarchy makes sense
- [ ] Each epic has clear user value
- [ ] Story outlines are comprehensive
- [ ] Dependencies are mapped
- [ ] Critical path identified

### Resources
- [ ] Team allocation realistic
- [ ] Key people assigned
- [ ] Timeline accounts for capacity

### Risk Management
- [ ] High-impact risks identified
- [ ] Mitigation strategies defined
- [ ] Contingency plans exist

### Alignment
- [ ] Plan aligns with PRD goals
- [ ] No conflicts with PRD requirements
- [ ] Terminology matches PRD

### Constitution Principles
- [ ] ✅ Each epic delivers user value
- [ ] ✅ Success metrics are measurable
- [ ] ✅ Plan is self-contained
- [ ] ✅ Work is properly sequenced
- [ ] ✅ Terms consistent with PRD

---

## Stories Checklist

### Story Structure
- [ ] Follows "As a [user], I want [capability], so that [benefit]" format
- [ ] User type is specific
- [ ] Capability is clear
- [ ] Benefit explains the "why"

### Acceptance Criteria
- [ ] Uses GIVEN-WHEN-THEN format
- [ ] Criteria are testable (objective, not subjective)
- [ ] Edge cases covered
- [ ] Error scenarios defined
- [ ] No vague language ("intuitive", "easy", "fast")

### Context
- [ ] Background provided
- [ ] Fits into epic/release context
- [ ] Related features mentioned

### Scope
- [ ] What's included is clear
- [ ] What's NOT included is explicit
- [ ] No scope creep from previous stories

### Dependencies
- [ ] Story dependencies identified
- [ ] Blocking issues noted
- [ ] Technical dependencies documented

### Technical Details
- [ ] Implementation guidance (not requirements) provided
- [ ] Performance needs stated if applicable
- [ ] Security considerations if applicable

### Size
- [ ] Story is appropriately sized (not too large)
- [ ] Can be completed in one sprint
- [ ] If XL, consider breaking down

### Constitution Principles
- [ ] ✅ User value is explicit
- [ ] ✅ Acceptance criteria are testable
- [ ] ✅ Story is self-contained
- [ ] ✅ Priority assigned
- [ ] ✅ Terms match PRD/Plan

---

## Quick Issues Checklist

### All Issue Types
- [ ] Clear, descriptive title
- [ ] Type correctly identified (Story/Bug/Task/Spike)
- [ ] Priority assigned with rationale
- [ ] Estimate provided
- [ ] Labels suggested
- [ ] Context sufficient for execution

### User Stories
- [ ] User value clearly stated
- [ ] Acceptance criteria testable
- [ ] Edge cases considered
- [ ] Dependencies identified
- [ ] Can stand alone or references existing context

### Bugs
- [ ] Reproducible steps clear
- [ ] Expected vs actual behavior documented
- [ ] Impact assessed
- [ ] Environment details included
- [ ] Severity appropriate

### Technical Tasks
- [ ] User/business value explained (even if indirect)
- [ ] Success criteria defined
- [ ] Deliverables specific
- [ ] Testing strategy included
- [ ] Rollback plan if needed

### Spikes
- [ ] Question to answer is specific
- [ ] Time-boxed appropriately
- [ ] Expected deliverables defined
- [ ] Links to follow-up decision
- [ ] Evaluation criteria clear

### Constitution Principles
- [ ] ✅ Value articulated (user or business)
- [ ] ✅ Success/acceptance criteria testable
- [ ] ✅ Sufficient context provided
- [ ] ✅ Priority clear
- [ ] ✅ Consistent terminology

---

## Cross-Document Consistency

### PRD ↔ Plan
- [ ] Plan goals align with PRD goals
- [ ] Plan epics cover all P0 requirements from PRD
- [ ] No conflicts between PRD scope and Plan scope
- [ ] Timeline in Plan is realistic given PRD complexity

### Plan ↔ Stories
- [ ] Stories cover all epics in Plan
- [ ] Story estimates align with Plan timeline
- [ ] Dependencies in stories match Plan dependencies
- [ ] Success metrics in stories support Plan metrics

### PRD ↔ Stories
- [ ] Stories deliver on PRD requirements
- [ ] User value in stories matches PRD goals
- [ ] No stories outside PRD scope
- [ ] Terms used consistently

---

## Language & Communication

### Clarity
- [ ] No jargon without definition
- [ ] Abbreviations explained on first use
- [ ] Sentences are concise
- [ ] Technical and non-technical readers can understand

### Terminology
- [ ] Same terms used throughout all docs
- [ ] Glossary exists if needed
- [ ] No synonyms for core concepts

### Completeness
- [ ] All sections filled out
- [ ] No "TBD" without owner and date
- [ ] Links work
- [ ] References are accurate

---

## Before Sharing

### Review
- [ ] Self-reviewed for typos and errors
- [ ] Peer reviewed (if applicable)
- [ ] Stakeholder feedback incorporated

### Versioning
- [ ] Version number assigned
- [ ] Last updated date is current
- [ ] Major changes documented

### Distribution
- [ ] Saved in correct location
- [ ] Shared with appropriate people
- [ ] Access permissions correct

---

## Multi-Language Support

This checklist applies to documentation in any language. Ensure:
- [ ] Language is consistent throughout document
- [ ] Headers and structure match template for that language
- [ ] Technical terms are correctly translated or kept in English
- [ ] Date formats are culturally appropriate
